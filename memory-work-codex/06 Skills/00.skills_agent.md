# Skills Library Agent

## Zone Purpose

**Skills** are executable capability modules—more powerful than static tools. A skill can:
- Run scripts or commands
- Generate outputs (calendars, reports, analyses)
- Trigger workflows or processes
- Manage system operations
- Integrate with external services

Skills bridge your knowledge system and actual automation.

## Pre-Installed Skills

### 1. datetime-check
**Purpose**: Calibrated local time retrieval
**Trigger**: Protocol-based (mandatory first-load every session)
**Function**: Returns corrected local time (Beijing/UTC offset), date, day of week, and calendar week number
**Used by**: Startup sequence to determine sync depth (light vs. deep)

### 2. ics-generator
**Purpose**: Calendar file generation
**Trigger**: Skill-based (user requests calendar export)
**Function**: Converts event data to ICS format; supports recurring events, reminders, categories
**Output**: `.ics` file (iCalendar format)

### 3. save-conversation
**Purpose**: Export conversation as markdown
**Trigger**: Skill-based (end-of-conversation export prompt)
**Function**: Converts chat history to markdown with metadata (date, length, key topics)
**Output**: `.md` file with YAML frontmatter

### 4. memory-review
**Purpose**: Weekly memory calibration and retrospective
**Trigger**: Skill-based (triggered by "memory-review" or weekly prompt)
**Function**:
- Scans MEMORY.md for items due for review
- Generates summary of past week's learning
- Prompts user to confirm/update memory entries
- Identifies items ready for graduation (MEMORY → USER)
**Output**: Updated MEMORY.md, summary report

### 5. iteration-log
**Purpose**: Architecture changelog management
**Trigger**: Skill-based (triggered when system changes occur)
**Function**: Records changes to knowledge structure (new zones, new skills, system updates)
**Output**: Updated CHANGELOG.md, summary of changes

## How to Add New Skills

### 1. Create Skill Folder
```bash
mkdir -p 06\ Skills/new-skill-name
```

### 2. Write SKILL.md
Include YAML frontmatter, purpose, trigger mechanism, inputs/outputs, and instructions.

### 3. Add Scripts (if needed)
Place executable code in `scripts/` subdirectory.

### 4. Add References (if needed)
Place supporting documentation in `references/`.

### 5. Register in CLAUDE.md
Add entry to skills list with trigger rule:
```yaml
- skill: new-skill-name
  trigger: "semantic match OR explicit call"
  category: "Automation / Data / Analysis / Integration / Workflow"
```

## Runtime Mechanisms

### Protocol (Reflexive)
- Defined in CLAUDE.md startup sequence
- Runs **automatically every session** without user request
- Examples: datetime-check, focus-zone sync
- Like reflexes: background, essential, non-negotiable

### Skill (Deliberate)
- Triggered by semantic match (AI interprets user intent)
- Examples: "save conversation", "generate calendar", "do a memory review"
- User doesn't need exact trigger words; intent-based
- Like conscious actions: requested, flexible, adaptive

## Agent Behavior in This Zone

- **Creating**: Propose new skill before writing SKILL.md (explain purpose, trigger)
- **Modifying**: Update existing skills; version them
- **Testing**: Verify new skills work before registering
- **Documentation**: Keep README.md updated with skill index
- **Deduplication**: Check for similar skills before creating new ones

## Best Practices

1. **Single Responsibility**: One skill = one capability
2. **Clear Trigger**: Define exactly when/how the skill runs
3. **Reusable Parameters**: Accept inputs, don't hard-code values
4. **Error Handling**: Graceful failures, informative errors
5. **Documentation**: Every skill needs clear purpose and usage
6. **Versioning**: Track changes; maintain compatibility
7. **Testing**: Skills should have examples in SKILL.md

## Organization Rules

- Alphabetical or functional grouping
- Clear naming: `skill-name` (kebab-case)
- Consistent file structure
- Keep `scripts/` separate from documentation
- Archive old skill versions with timestamp
